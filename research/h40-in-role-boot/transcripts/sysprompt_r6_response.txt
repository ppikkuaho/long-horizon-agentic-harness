❯ Build a URL shortener.

⏺ Intent

  Build a URL shortener service that maps short codes to long URLs and redirects visitors.

  ---
  Clarifying Questions

  Before decomposition is sound, I need answers to these load-bearing unknowns:

  Scale & Deployment
  1. Expected request volume? (personal/hobby vs. production traffic)
  2. Where does this run — local only, a single VPS, serverless (Vercel/Cloudflare Workers), or a cloud provider?

  Persistence
  3. What data store is acceptable? (SQLite file, Postgres, Redis, in-memory only, managed DB)
  4. Do short links expire, or are they permanent?

  Auth & Access Control
  5. Is link creation open to anyone, or gated behind an API key / login?
  6. Is there an admin UI, or is management purely API/CLI?

  Short Code Generation
  7. Custom aliases allowed (e.g. /my-link), or auto-generated only?
  8. Any vanity/reserved-word list (e.g. block /admin, /api)?

  Analytics
  9. Track click counts? Referrers? Geo? Or no analytics?

  Non-functionals
  10. Any language/framework preferences or hard constraints?
  11. Is a web UI required, or is a pure API sufficient?

  ---
  First-Cut Area Decomposition

  Even before your answers, here is the natural seam structure to react to:

  ┌─────────────────────────┬──────────────────────────────────────────────────────────────────────────┐
  │          Area           │                              Responsibility                              │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ A — Short Code Engine   │ Generate/validate short codes, enforce uniqueness, handle custom aliases │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ B — Storage Layer       │ Persist URL ↔ code mappings, TTL/expiry, read-through cache              │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ C — Redirect Service    │ Accept GET /<code>, resolve to long URL, issue HTTP 301/302, log hit     │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ D — Link Management API │ Create, read, delete links; return short URL; enforce auth               │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ E — Analytics Collector │ Record click events (count, timestamp, referrer, UA); expose aggregates  │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ F — Auth & Access       │ API key issuance, middleware guard on write endpoints                    │
  ├─────────────────────────┼──────────────────────────────────────────────────────────────────────────┤
  │ G — Web UI (optional)   │ Form to shorten a URL; dashboard to list/manage links                    │
  └─────────────────────────┴──────────────────────────────────────────────────────────────────────────┘

  Likely sequencing (areas unlock each other):
  A (codes) → B (storage) → C (redirect) → D (API) → F (auth) → E (analytics) → G (UI)

  Highest-uncertainty risks:
  - Redirect latency — storage choice dominates; wrong pick here affects everything downstream
  - Code collision — short code namespace and uniqueness strategy must be locked in Area A before B and C touch it
  - Analytics write path — synchronous vs. async recording on hot redirect path is a key architectural fork

  ---
  Please answer as many of the clarifying questions as you can, and confirm whether the area decomposition looks right. Once I have those answers I'll write scoped briefs for each L3 Module Designer.
