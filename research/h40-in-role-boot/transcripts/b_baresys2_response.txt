❯ Build a URL shortener.

⏺ Intent (restated)

  Design a URL shortener: a service that accepts long URLs and returns short aliases that redirect to the originals.

  That phrase hides a lot of decisions. Before I decompose, I need you (or the orchestrator above me) to answer the load-bearing questions below. The architecture changes materially depending on the answers.

  ---
  Clarifying questions (answer these before decomposition is sound)

  1. Scope & surface area
  - Is this a public multi-tenant service (think bit.ly), an internal tool, or a library/embed?
  - Web UI, JSON API, CLI — which surfaces are in scope?
  - Anonymous use allowed, or auth-gated? If auth: who owns the identity (built-in, SSO, OAuth)?

  2. Scale & SLOs
  - Expected write rate (links/sec created) and read rate (redirects/sec)?
  - Target p99 redirect latency? Availability target?
  - Geographic distribution — single region or global edge?

  3. Slug semantics
  - Random short codes, or user-chosen custom aliases, or both?
  - Required length / alphabet? Case-sensitive?
  - Are slugs guessable acceptable, or must they be unpredictable (security concern for private links)?
  - Idempotency: does the same long URL always map to the same slug, or always a new one?

  4. Lifecycle
  - Expiration (TTL, max-uses, never)?
  - Editable destinations after creation? Deletable? Soft-delete vs hard?
  - Rate limiting / abuse controls (malware/phishing scanning, domain blocklists)?

  5. Analytics
  - Per-link click counts? Per-click metadata (timestamp, referrer, UA, geo)?
  - Real-time vs batch? Retention window?

  6. Persistence & ops
  - Existing DB/infra preference, or greenfield?
  - Deployment target (container, serverless, single-binary on a VM)?
  - Backup/restore requirements?

  7. Non-functionals
  - Language/runtime constraint?
  - Open-source license, observability stack, deadline?

  ---
  First-cut area decomposition (for you to react to)

  Assuming a modest, auth-optional JSON+web service backed by a single relational store, I'd hand down roughly these L3 briefs:

  ┌─────┬────────────────────┬─────────────────────────────────────────────────────────────────────────────────────────┬────────────────────────────────────┐
  │  #  │        Area        │                                   Core responsibility                                   │            Key contract            │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ A   │ Slug Service       │ Generate, validate, and reserve short codes; enforce alphabet/length/collision policy   │ reserve(longUrl, opts) -> slug     │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ B   │ Link Store         │ Persist (slug → longUrl + metadata); enforce uniqueness; support lookup, update, delete │ CRUD over Link aggregate           │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ C   │ Redirect Endpoint  │ Hot path: resolve slug → 301/302/307 to destination; emit click event                   │ GET /{slug}                        │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ D   │ Management API     │ Create/list/update/delete links; auth & ownership checks                                │ POST/GET/PATCH/DELETE /links       │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ E   │ Analytics Pipeline │ Capture click events, aggregate, expose stats                                           │ event in → counters/timeseries out │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ F   │ Abuse & Safety     │ URL validation, blocklist, optional safe-browsing check, rate limits                    │ validate(longUrl) -> ok | reason   │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ G   │ Web UI (optional)  │ Minimal front-end for create/manage/view stats                                          │ consumes Management API            │
  ├─────┼────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────┼────────────────────────────────────┤
  │ H   │ Platform Concerns  │ Config, logging, metrics, deploy artifact, migrations                                   │ cross-cutting                      │
  └─────┴────────────────────┴─────────────────────────────────────────────────────────────────────────────────────────┴────────────────────────────────────┘

  Sequencing (highest-uncertainty first):
  1. Lock answers to §1–§3 above — they constrain everything else.
  2. A + B + C as the minimum redirect loop (the thing that must be correct and fast).
  3. D + F so creation is safe.
  4. E and G as enhancements.

  Top risks to flag now:
  - Hot-path latency — redirect must be cache-friendly; choice of slug generation and store shape determines this.
  - Slug collisions vs. predictability — random short codes trade collision retries against guessability.
  - Abuse vector — open shorteners become phishing/malware laundromats fast; safety is not optional if public.
  - Analytics write amplification — naïvely writing a row per click will dominate the workload.
