❯ Build a URL shortener.

⏺ Intent: Build a URL shortener service that maps short codes to long URLs and redirects users.

  Before decomposition is sound, I need answers to these load-bearing questions:

  Scale & deployment
  1. Expected request volume — hobby/personal, small team, or production-scale (millions of redirects/day)?
  2. Hosted where — local dev only, a VPS, a cloud provider (AWS/GCP/etc.), or serverless?

  Persistence
  3. Where do mappings live — in-memory (restart = data loss), SQLite, Postgres, Redis, or a managed DB?
  4. Are short codes permanent, or do they expire?

  Access & auth
  5. Is link creation open to anyone, or gated behind an API key / login?
  6. Do you need a web UI for creating links, or is a REST/CLI API enough?

  Short code design
  7. Random codes (e.g. x7kQ2), human-readable slugs, or custom aliases users choose?
  8. Any vanity/custom domain requirement (e.g. go.myco.com/foo)?

  Analytics
  9. Do you need click tracking — total counts, referrer, geo, timestamps?

  Tech constraints
  10. Language/framework preference, or greenfield choice?

  ---
  First-cut area decomposition (pending your answers):

  ┌────────────────────────────┬───────────────────────────────────────────────────────────────────────────────────┐
  │            Area            │                                  Responsibility                                   │
  ├────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Ingestion API              │ Accept a long URL, validate it, generate/store a short code, return the short URL │
  ├────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Redirect Engine            │ Receive a short code, look up the long URL, issue a 301/302, handle misses        │
  ├────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Storage Layer              │ CRUD for code→URL mappings; abstracts the backing store                           │
  ├────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Code Generation            │ Produce collision-resistant short codes (random, hash, or custom)                 │
  ├────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Analytics Sink (optional)  │ Record click events asynchronously; query interface for counts                    │
  ├────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Auth/Rate-limit (optional) │ API key validation, per-key quotas, abuse prevention                              │
  ├────────────────────────────┼───────────────────────────────────────────────────────────────────────────────────┤
  │ Web UI (optional)          │ Form to shorten a URL; dashboard for link management                              │
  └────────────────────────────┴───────────────────────────────────────────────────────────────────────────────────┘

  The critical path is Storage → Code Generation → Ingestion API → Redirect Engine in that order. Analytics and Auth are addable after the core works.

  Answer the questions above and I'll produce tight briefs for each module.
