❯ Build a URL shortener.

⏺ Intent

  Build a URL shortener service that maps short codes to destination URLs and redirects users.

  ---
  Clarifying Questions

  Before decomposition is sound, I need answers to these load-bearing questions:

  Scale & Deployment

  1. Traffic expectations — personal/hobby tool, small team, or public-facing at scale? (This drives caching, rate limiting, database choice.)
  2. Deployment target — local dev only, a specific cloud provider (AWS/GCP/Vercel/Fly.io), self-hosted VPS, or containerized?

  Interface

  3. API-only, or does it include a UI? — REST API only, or also a web UI for creating/managing links? (A dashboard? A simple form?)
  4. CLI client? — Should users be able to shorten URLs from the terminal?

  Persistence & State

  5. Database preference — SQLite (simple), Postgres, Redis, or in-memory (ephemeral/demo only)?
  6. Persistence durability — Is data loss on restart acceptable?

  Features & Auth

  7. Authentication — Anonymous creation (anyone can shorten), or does it require accounts/API keys?
  8. Custom slugs — Can users specify their own short code (e.g. /my-link), or auto-generated only?
  9. Analytics — Track click counts? Referrers? Geo? Or none?
  10. Expiry — Should links expire (TTL or date-based)?
  11. Link management — Does someone need to list, edit, or delete their links?

  Tech Stack

  12. Language/framework preference — Python (FastAPI/Flask), Node (Express/Hono), Go, other?

  ---
  First-Cut Area Decomposition

  Assuming a minimal but production-shaped service (REST API + redirect, persistent storage, no auth, no UI), here are the natural module boundaries:

  ┌─────────────────────────────────────────────────────────┐
  │  A. API Layer                                           │
  │     POST /shorten → { short_url }                       │
  │     GET  /:code   → 301/302 redirect                    │
  │     GET  /:code/stats (optional)                        │
  ├─────────────────────────────────────────────────────────┤
  │  B. Short Code Generator                                │
  │     Collision-safe code generation strategy             │
  │     Custom slug validation (if supported)               │
  ├─────────────────────────────────────────────────────────┤
  │  C. Storage / Repository Layer                          │
  │     CRUD for (code ↔ destination URL) mappings          │
  │     TTL / expiry enforcement (if supported)             │
  │     Click event recording (if analytics needed)         │
  ├─────────────────────────────────────────────────────────┤
  │  D. Redirect Engine                                     │
  │     Lookup short code → destination                     │
  │     Choose 301 (permanent) vs 302 (temporary)           │
  │     Cache layer (if scale warrants it)                  │
  ├─────────────────────────────────────────────────────────┤
  │  E. Infrastructure / Configuration                      │
  │     DB migrations / schema                              │
  │     Env config, secrets management                      │
  │     Containerization / deployment manifests             │
  ├─────────────────────────────────────────────────────────┤
  │  F. Web UI (optional)                                   │
  │     Form to shorten a URL                               │
  │     Link list / management view                         │
  └─────────────────────────────────────────────────────────┘

  Sequencing risk order: C (storage contract) must be locked before A and D can be fully specced. B is independent. F is additive and can be deferred.

  ---
  Answer those questions and I'll tighten the decomposition and issue briefs to the Module Designers below.
