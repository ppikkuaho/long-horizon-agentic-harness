❯ Build a URL shortener.

⏺ Intent

  Build a URL shortener service that maps short codes to long URLs and redirects users accordingly.

  ---
  Clarifying Questions

  Before decomposition is sound, I need answers to these load-bearing unknowns:

  1. Scope & Users

  - Is this a public service (anyone can shorten URLs) or private/authenticated (logged-in users only)?
  - Expected scale: personal tool, team tool, or production-grade public service (affects storage and infra choices)?

  2. Persistence

  - Where should shortened URLs be stored — a proper database (Postgres, SQLite), a hosted key-value store (Redis), or is in-memory acceptable (dev/demo)?
  - Do short links need to survive restarts?

  3. Short Code Generation

  - Random codes (e.g. abc123) or custom aliases (user picks my-link), or both?
  - Any required code length or character set constraints?

  4. API Surface

  - Web UI (browser form), REST API only, or both?
  - Should there be a dashboard to list/manage your links?

  5. Non-Functionals

  - Analytics: track click counts, referrers, geolocation?
  - Expiry: do links expire (TTL)?
  - Rate limiting / abuse prevention?
  - Auth: if yes, which mechanism — sessions, API keys, OAuth?

  6. Deployment Target

  - Where does this run? Local dev only, containerized (Docker), a specific cloud/PaaS, or serverless?
  - Language/runtime preference (Node, Python, Go, etc.)?

  ---
  First-Cut Area Decomposition (provisional — subject to answers above)

  ┌────────────────────────────────┬──────────────────────────────────────────────────────────────────────────────────────┐
  │              Area              │                                    Responsibility                                    │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ A. Short Code Engine           │ Generate/validate short codes; collision handling; optional custom aliases           │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ B. Storage Layer               │ CRUD for {short_code → long_url, metadata}; abstracted behind a repository interface │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ C. Redirect Service            │ Receive GET /{code}, resolve URL, emit 301/302, handle misses                        │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ D. Ingestion API               │ Accept long URL, invoke Code Engine, persist, return short URL                       │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ E. Auth & Identity (if needed) │ Protect ingestion; associate links to users                                          │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ F. Analytics (if needed)       │ Record click events; aggregate counts/referrers                                      │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ G. Admin / UI (if needed)      │ Web form for shortening; dashboard for listing/managing links                        │
  ├────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ H. Infra / Config              │ Environment config, deployment packaging, rate limiting                              │
  └────────────────────────────────┴──────────────────────────────────────────────────────────────────────────────────────┘

  Critical path (minimum viable): B → A → D → C — storage, then code generation, then create endpoint, then redirect endpoint. Everything else layers on top.
