❯ Build a URL shortener.

⏺ Intent

  Build a URL shortener service that maps short codes to long URLs and redirects users accordingly.

  ---
  Clarifying Questions

  Before decomposition is sound, I need answers to these load-bearing unknowns:

  Scale & Deployment

  1. Traffic profile — personal/hobby tool, small team, or public-facing at scale? (This drives whether we need a cache layer, CDN, or just a single process.)
  2. Hosting target — local only, a specific cloud provider, serverless (e.g. Cloudflare Workers), or containerized?

  Data & Persistence

  3. Persistence — ephemeral (in-memory, restarts wipe links), or durable (database)? If a DB, any preference (SQLite, Postgres, Redis)?
  4. Link expiry — do short links ever expire? TTL-based, click-count-based, or never?

  Auth & Access Control

  5. Who can create links? — anonymous (anyone), authenticated users only, or admin-only?
  6. Multi-tenancy — do different users own different links, or is it a shared namespace?

  Features

  7. Custom slugs — can users choose their own short code (e.g. /my-brand), or is it always auto-generated?
  8. Analytics — track click counts? Referrer, geo, timestamps? Or no analytics at all?
  9. API vs. UI — REST/JSON API only, a web UI only, or both?
  10. Link management — do creators need to list, edit, or delete their own links?

  Non-Functionals

  11. Short code length/alphabet — any constraints (e.g. alphanumeric only, 6 chars)?
  12. Vanity domain — does the shortened link use a custom domain (e.g. go.example.com)?

  ---
  First-Cut Area Decomposition

  Assuming a modest-scale web service with durable storage, here's how I'd carve it up. React to this and the answers above will sharpen the boundaries:

  ┌─────────────────────────────────────────────────────┐
  │                   A. API Gateway                    │
  │  Routing, rate-limiting, request validation         │
  └────────────┬────────────────────────┬───────────────┘
               │                        │
  ┌────────────▼──────────┐  ┌─────────▼───────────────┐
  │  B. Redirect Engine   │  │  C. Link Management API  │
  │  GET /{code} → 301/   │  │  Create / read / delete  │
  │  302 to long URL      │  │  short links             │
  └────────────┬──────────┘  └─────────┬───────────────┘
               │                        │
  ┌────────────▼────────────────────────▼───────────────┐
  │              D. Code Generation Service             │
  │  Produce unique, collision-free short codes         │
  └─────────────────────────────┬───────────────────────┘
                                │
  ┌─────────────────────────────▼───────────────────────┐
  │                  E. Data / Storage                  │
  │  Link records (code ↔ URL, metadata, TTL)          │
  └─────────────────────────────────────────────────────┘

  Optional modules (answers-dependent):
    F. Auth / Identity      — if authenticated users
    G. Analytics Collector  — if click tracking needed
    H. Web UI / Frontend    — if browser interface wanted

  Key contracts between areas:
  - B → E: read-heavy path — lookup(code) must be fast (cache candidate)
  - C → D → E: write path — generate code, persist atomically, return short URL
  - F → C: authz guard — link creation/deletion gated on identity
  - G ← B: async event on every redirect (must not slow B down)

  ---
  Please answer the questions above (even roughly — "hobby project, SQLite, no auth, just counts" is enough) and I'll sharpen the module boundaries and hand scoped briefs down to the Module Designers.
