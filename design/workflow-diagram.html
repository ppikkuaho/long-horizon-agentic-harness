L1-L4 Workflow

Sequence-style swimlane -- follow the arrows from Request to Deliverable
UserHuman
1Request
User submits a request or task. This is the entry point for all work in the system.
4Confirm / correct
User validates L1's understanding of intent, or provides corrections to ensure alignment before work begins.
27Receive deliverable
User receives the final shaped output. The complete round-trip from request to delivery is finished.
L1Orchestrator
2Intent reading
L1 analyzes the user's request to understand true intent, context, and implicit requirements beyond what was literally stated.
3Verify with user
L1 presents its understanding back to the user for validation. Catches misalignment early before any planning begins.
5Route to L2
L1 packages the confirmed intent and routes it to L2 for planning and decomposition.
◆11Pre-execution gate
GATE: L1 reviews L2's proposed approach. Must sign off before any execution begins. Can reject, request changes, or approve.
26Shape delivery
L1 shapes the raw results into a user-appropriate deliverable. Translates technical output into user-facing form.
L2Planner
6Generative skeleton
L2 creates a high-level structural plan -- the skeleton that will be filled in by domain experts.
7Spawn planning L3s
L2 spawns temporary L3 agents to decompose work across relevant domains in parallel.
9Cross-domain review
L2 reviews domain decompositions for conflicts, dependencies, and coherence across all domains involved.
10Present approach
L2 presents the consolidated plan to L1 for approval at the pre-execution gate.
12Write L3 briefs
After gate approval, L2 writes detailed execution briefs for each L3 domain agent with scope, constraints, and objectives.
◆14Plan sign-off
GATE: L2 reviews L3 decompositions and acceptance tests. Signs off on the execution plan before L4 work begins.
24Evaluate results
L2 evaluates integrated results against the original plan and acceptance criteria. Checks for completeness and quality.
25Report to L1
L2 reports evaluation results to L1 with summary, outcomes, and any deviations from the original plan.
L3Domain Lead
8Domain decomposition
Parallel temporary L3s decompose work within their domains. They identify tasks, dependencies, constraints, and domain-specific requirements.
13Decompose + tests
L3 decomposes briefs into concrete tasks with acceptance tests. Defines precisely what "done" looks like for each task.
15Spawn L4s
L3 spawns L4 execution agents with specific task assignments, acceptance criteria, and context needed to do the work.
22Review L4 reports
L3 reviews completed L4 work reports, checking results against acceptance criteria and domain standards.
23Integrate domain
L3 integrates all L4 results within its domain and reports the consolidated outcome up to L2.
L4Executor
16Execute task
L4 performs the actual work -- coding, writing, analysis, research, or whatever the task requires.
17Self-verify
L4 runs its own verification against acceptance tests before submitting. Catches obvious issues early before review.
18Write report
L4 documents what was done, decisions made, trade-offs considered, and any issues encountered. Submitted for review.
ReviewQuality Dept
19Automated tools
Automated quality checks: linting, type checking, test suites, security scanning, and any configured CI pipeline steps.
20Code review
Structured code/work review against acceptance criteria, coding standards, architectural principles, and best practices.
◆21Review verdict
GATE: Pass or fail. Pass sends work to L3 for integration. Fail kicks back to L4 with specific feedback for rework.
