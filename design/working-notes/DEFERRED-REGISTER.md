# Deferred / Owed / Open Register

**What this is.** The single consolidated ledger of every concern that is *not* fully built in the v1 walking skeleton: things owed, things deliberately deferred behind a trigger, and open forks/dependencies that still need a decision. This is the other half of the completeness pair (`design/INDEX.md` references it). **If an owed/deferred/open concern is not in this register, it is presumed dropped.** This register is the home-of-last-resort named by the homeless-concern check (Lesson 3): a concern with no design cluster and no build increment lives *here* until it gets one.

This register does **not** restate the build sequence — `path-to-ready-2026-06-04.md` (the 8 phases) and `harnessd/IMPLEMENTATION-PLAN.md` (Increments 0–16) own that. This register collects the *gaps in* that plan: what the v1 cut leaves unbuilt, and the trigger or owner that brings each item back.

## MAINTENANCE RULE (owner + update rule, per Lesson 1)

- **Owner:** whoever lands a design or build change that defers, owes, or opens a concern.
- **Rule:** a new owed/deferred/open item is added to this register **in the same change** that creates it. A deferral that isn't recorded here is a silent drop. When a deferred item is pulled in (its trigger fires), strike it through here with the increment/commit that closed it — don't delete the row (traceability).
- **Canon-precedence:** per `working-notes/INDEX.md`, the LIVE Jun-04 decision notes override the Jun-02 specs where they conflict. This register inherits that precedence; where a row cites both a spec and a Jun-04 note, the Jun-04 note wins.

## METHODOLOGY LESSONS (durable capture — do not re-learn)

1. **A doc-MAP needs an owner + an update rule or it goes stale and misleads.** The `design/INDEX.md` staleness "lost" the cluster specs (the auditor trusted the map and reported the specs absent). The fix: an index/map carries a maintenance rule *in the index itself*, and an owner who updates it in the same change. This register's MAINTENANCE RULE above is the application of that lesson.
2. **Absence/completeness audits CONFABULATE absence.** An auditor handed only a map will hallucinate that real files are missing. Never let an auditor trust a map: hand it the actual corpus, OR require an `ls`-and-`grep`-and-read verify of every claimed-absent file. This register applied the rule — every spec-premised "absent" row below was grepped + read against `DAEMON.md` / `WATCHDOG.md` / `IMPLEMENTATION-PLAN.md`, not inferred from an index. Natural home: `DECOMPOSITION-METHODOLOGY.md`.
3. **The HOMELESS-CONCERN check.** For every concern, name the ONE artifact (design cluster / build increment / this register) that owns it, AND say what happens to it across (a) a code upgrade, (b) project end-of-life, (c) the human boundary. If you can't point to a line, the concern is homeless — and homeless concerns land *here* by default. Every row below carries an explicit `where-the-design-lives` per this rule.
4. **Confabulation recurs even INSIDE a confabulation-aware audit (4th instance) — for absence/staleness questions, the orchestrator READS ground truth itself; a subagent's "I verified the actual files" is NOT trustworthy.** The 2026-06-05 role-docs reconciliation audit (task `wz4ecm1pw`) claimed line-by-line file verification and returned "significant-drift": ~5 build-blockers premised on terminal-signal / liveness / sign-off content being ABSENT from `comms-protocol.md`, `agent-lifecycle.md`, and the L5 docs, plus naming-sweeps for "Internal Affairs" ×4 and a stale "status/ board" line. Ground-truth grep + read showed every one of those sections **exists and is well-formed** (terminal-signal `comms-protocol §134–148`; liveness/sign-off-or-fail `agent-lifecycle §117–127`; direct-children/whole-subtree split `§131–135`; L5 sign-off `role §45` + `spawn-template step 11`), and "Internal Affairs" / "status/ board" have **zero** occurrences. The audit confabulated phantom ABSENCES *and* phantom PRESENCES, and misreported line-counts (`comms-protocol` "138 lines" — actually 218) while citing §-numbers that only fit the true length. Exactly **one** real gap survived verification: `agent-definition-principles §4` stale H40 framing ("future patch / compensate in prose") → reframed to the resolved model (one shared minimal `--system-prompt-file`; role-as-documents). **Cure, sharper than Lesson 2:** a delegated absence-audit gates *direction-of-attention only* — never the edit-list. Before any edit premised on "X is missing/stale," the orchestrator greps + reads the corpus itself. A subagent self-reporting "I verified the real files" is as confabulation-prone as the drift it audits. Natural home: `DECOMPOSITION-METHODOLOGY.md`.
5. **Fork-handling in the build phase: decide spec-faithfully, note, proceed — don't block.** (User directive, 2026-06-05.) For a fork encountered while building, *unless it DIVERGES from the spec*: exercise best judgment for the most spec-faithful resolution, **annotate the decision** (a `RESOLVED` marker in the owning doc + a row in `working-notes/FORK-DECISIONS.md`), and **proceed**. Do NOT stop to ask. **Present the accumulated fork decisions to the user when the build is finished.** Stop ONLY on an *actual blocker*: a genuine spec ambiguity/divergence, a test that cannot be made green, or a missing dependency. (Supersedes the earlier "stop on any fork" build-cadence framing — that over-stopped on forks the spec already answers.)
6. **A mocked boundary is an UNVALIDATED ASSUMPTION until a contract test pins the mock to the real implementation.** (User challenge, 2026-06-06, Increment 6.) Mocking tmux in the detector (Increment 6) is correct for unit-testing the *fusion logic* (the verdict rules + the within-W false-idle hazard are timing-dependent and can't be tested deterministically against a live pane). BUT a mock-only suite proves only "the detector reasons correctly GIVEN what tmux returns" — it proves nothing about whether real tmux returns that. The cure is a **contract test**: somewhere the SAME signal functions (`pane_alive`, `jsonl_progress`) must run against the REAL boundary — a real living pane, a real `kill`'d (dead) pane, a real growing/flat transcript — pinning the exact behavior the mock assumes, so the two can't drift. **Concrete commitment:** Increment 9 (concrete `tmux.py`, real local tmux server + fake CLI pane, zero model burn) MUST add a pane-liveness/jsonl contract test exercising the real `detector_signals` against real tmux, not only the OAuth env-leak test it currently specs. General rule: every mocked dependency names where its contract is validated against reality, or the mock is a liability. Natural home: `DECOMPOSITION-METHODOLOGY.md`.
8. **Read-side torn-tail tolerance is necessary but NOT sufficient — recovery that APPENDS must first durably HEAL (compact) the torn tail, or its own writes turn a torn TAIL into a torn MIDDLE and brick every subsequent load.** (Found by Increment 15's REAL kill-9, 2026-06-06 — the single most valuable catch of the build, and proof of the bias-to-real rule.) Increment 2 made `load_wal` torn-tail-tolerant ON READ (truncate-and-continue), and every Inc-2 test passed — but those tests NEVER appended a real frame after a torn tail. A real `SIGKILL` leaves an unterminated partial frame ON DISK; `store.append_framed` opens in `"a"` mode, so the FIRST recovery append (the necro) concatenates onto the torn bytes and supplies their missing `\n`, freezing them as a permanent NON-final corrupt line; the SECOND recovery append (the L1 claim) then makes the very next `load_wal` (via `next_seq`/`validate`) FAIL CLOSED with `WALCorruptionError` — recovery bricks. The §4.4 "only the FINAL line can ever be torn" invariant held WITHIN one append but was VIOLATED ACROSS appends. **Fix:** `ledger._heal_torn_tail(path)` at the top of `append_wal` — if the WAL's last byte is not `\n`, truncate the unterminated trailing frame back to the last clean newline BEFORE appending (the dropped bytes are intent-first-uncommitted, exactly what `load_wal` drops on read). Verified: reverting the heal bricks the kill-9 gate; the heal is a strict no-op on a clean WAL with zero committed-state loss. **The general lesson:** a read-side crash-tolerance property must be re-proven across the WRITE path that follows recovery — and ONLY a real crash + real multi-append recovery surfaces it (a simulated/in-process crash or a read-only torn-tail test cannot). This is why the durability GATE is a real `os.kill(SIGKILL)`, not a simulation. Natural home: `DAEMON.md §4.4`.
7. **STANDING BUILD RULE — bias tests to REAL conditions; thoroughness over speed.** (User directive, 2026-06-06.) "Testing is cheap, fixing things after is expensive." For every remaining increment: the tester's DEFAULT is real conditions — real files, real processes, real `fork`/`fcntl`, real subprocess/tmux, real on-disk bytes — NOT mocks. A mock is permitted ONLY when there is a genuine, stated reason: (a) deterministic isolation of timing-dependent logic that cannot be exercised reliably against the real thing (e.g. the within-W false-idle hazard in Inc 6), (b) avoiding real model-token burn (deferred to the ONE real boot, Inc 16, behind `@pytest.mark.real_boot`), or (c) a runtime not yet built (the Codex adapter port). EVERY permitted mock must (i) keep its mock surface MINIMAL (mock only the unbuildable/timing/cost edge, leave everything else real — e.g. Inc 6 mocks only the tmux pane query, `jsonl_progress` hits a real file), and (ii) NAME where its contract is validated against reality (Lesson 6). When in doubt, choose the more-real, slower test. This rule is baked into every remaining increment's tester prompt.

## CAPTURED DECISIONS (2026-06-05)

These two are conversation decisions not yet written into any source doc. They are surfaced here as their own subsections (not buried in a table) because each reframes existing spec text.

### A. Security containment floor = v1 (minimal, bounded, NOT gold-plated)

The v1 security floor is four bounded pieces — effective but deliberately not gold-plated:

1. **A trust-boundary doc** — names what is trusted, what each spawned session may touch, and the accepted-risk posture for the current trusted dev box.
2. **A filesystem WRITE-JAIL** — each node's *writes* are confined to its own node workspace. Harness code, the binding-ledger, secrets, and `~` are **unreachable-for-write**. Crucially it is a write-jail, **not a read-jail**: reads stay allowed (per F34), so role-doc references still resolve read-only against the harness docs — no inline-flatten needed (see Decision B).
3. **A DECIDED spawned-session permission posture** — the permission mode the spawned agents boot under is an explicit decision, not an accident of the harness default.
4. **A fleet HALT kill-switch** — one operator action stops the whole fleet.

**Status:** v1-todo. **Where it lives:** a NEW security/trust-boundary design cluster owns this (currently homeless → owned by this register until that cluster doc exists); a v1 build increment to be added to `harnessd/IMPLEMENTATION-PLAN.md`. **Verified:** absent — no write-jail, trust-boundary doc, or fleet kill-switch anywhere in DAEMON / WATCHDOG / IMPLEMENTATION-PLAN. The fleet kill-switch is **co-owned** with the human-control write-verbs row (V12) in the table below.

### B. Role-file delivery REFRAME (supersedes the audit's "compose into one --system-prompt-file" framing)

The audit framed role delivery as "compose all role docs into one `--system-prompt-file`." That framing is **superseded**. The corrected model splits the system prompt from the role:

- **The system prompt stays MINIMAL and is SHARED across all levels** (one file, no per-level customization — there is no reason to fork it): the harness toolset + a bare operating posture common to every agent. It strips the base SWE/coding-assistant framing for two reasons: token-saving and no-out-of-role-pull. It is **not** minimal in order "to make the role bite."
- **The actual ROLE is delivered as DOCUMENTS** — each level's `soul + role + config + handbook` bundle is uploaded into the agent's context as role instructions, read like its brief. H40 confirms role-content *outside* the system prompt bites hard (CLAUDE.md overrode the base framing 6/6).
- **The write-jail (Decision A, write-only) lets the role-doc REFERENCES resolve** via read-only access to the harness docs. No inline-flatten of the bundle is required.
- **v1 work** = write the **one shared** minimal system prompt (a single file for all levels) — *to be drafted and shown to the user before anything uses it* — + make each level's role bundle readable from the node context. This is the reframed "role-resolution increment."

**Status:** DONE (2026-06-05). **Delivered:** (1) the one shared minimal system prompt promoted to `operational/shared/system-prompt.md`; (2) `agent-definition-principles §4` reframed; (3) the canonical bundle-delivery spec authored at `design/ROLE-RESOLUTION.md` (the split + the per-seat load-manifest + read-in-place reference resolution + chokepoint wiring + preserved invariants); (4) Decision B propagated across the corpus — DAEMON §6.2/§6.1-STEP2/§6.4/§7 + §3.2 binding (`role_file` → `system_prompt_file` constant + `role_variant` per-seat), SECURITY §1.1/§1.4/§2.3/§4.1/§9, IMPLEMENTATION-PLAN adapter+genesis+test-battery, and the spawn-template/intake load-manifests (completed with the level-scoped `shared/*` contract docs). **Verified:** adversarial-reviewed + orchestrator ground-truth grep (no residual `--system-prompt-file`→`role.md`, OAuth/write-jail/read-allow invariants intact); one load-scope over-claim (agent-definition-principles is L1–L4 not all-levels) caught + fixed in ROLE-RESOLUTION §2 + SECURITY §1.4. **Open seam (deferred, NOT blocking):** `role_bundle_hash` fencing field (DAEMON §3.2/§10, O9) — drift-detection over the role bundle; policy unresolved. **Where it lives:** `design/ROLE-RESOLUTION.md` + the reconciled DAEMON/SECURITY/IMPLEMENTATION-PLAN sections.
