# Architecture Gap Review — Runtime Model — 2026-06-04

**Source:** adversarial-review workflow `wf_f9047c79-a99` (7 lenses + synthesis, cross-checked against repo docs).
**Caveat:** the `state-ledger-races` lens stalled on all attempts and is **MISSING** — ledger write-races / atomicity / locking is **uncovered** here and needs a re-run. Run was heavy (~100 min, ~1M tokens, many retries).

**Headline:** the model is well-specified at the **semantic layer** (escalation payloads, sign-off content, visibility graph, addressing, the ephemeral/persistent split, agent-level cold-respawn recovery — several independently probed-and-cleared) but almost entirely unspecified at the **mechanism layer** that the locked interactive-tmux-TUI runtime just introduced. *"The parts of the system that aren't agents are the parts that aren't designed."* `tmux` and `watchdog` appear **zero times** in design/operational docs.

---

## BLOCKING for v1 (decide before a thin end-to-end build)

1. **The always-on harness daemon is asserted but never defined.** The runtime leans on a "deterministic always-on harness" (hosts watchdog, reaper, spawn script, ledgers; "infrastructure, not an agent" — agent-lifecycle.md:98, OBSERVABILITY.md:100) but no doc says what the process IS, who starts it, one-central vs one-per-node, or what happens when IT dies. Machine reboot/OOM kills the tmux server + every session + the watchdog together, with no surviving actor to recover. **Genesis** equally absent: every "bootstrap" is child-spawned-by-parent; L1 has no parent and nothing starts it or the daemon at first boot. *Deepest hole — nothing runs without it.*
   → **Resolved:** a named daemon spec started by a real service manager (launchd/systemd user unit, `Restart=always` = watchdog-of-the-watchdog) with PID/lock file; a **reconciliation-on-restart** routine (read binding ledger, check which uuids are alive in tmux, necro/mark-dead the rest); a **genesis** sequence (start daemon → spawn L1 in-role → register L1 as root node → resume-not-double-spawn on restart).

2. **Persistent-coordinator crash-recovery + completion-detection undefined; L5's terminal signal doesn't exist.** (a) agent-lifecycle.md:104-125 assumes a *live* parent notices a dead child — fine for L5, broken for a persistent L2/L3/L4 that dies while its own parent is parked. Under decision #5 a **dead coordinator with live children is indistinguishable from a healthy-waiting one** → orphaned work. (b) "Parent KILLS a child coordinator when done" has no mechanism — completion is only a best-effort bus nudge, no durable artifact. (c) The `DONE/FAILED/ESCALATED` tokens **exist nowhere in the L5 docs** (spawn-template.md:100 says "after you signal complete" but only report.md + a nudge are defined) — the watchdog's "is the report present" check has no defined target.
   → **Resolved:** process-liveness probe on coordinators (`pane_pid` death → "coordinator-died" event regardless of subtree activity → necro-from-ledger; dead-pid-but-live-children = recoverable orphan, not healthy-waiting); a durable coordinator-completion row (harness-stamped) the parent's kill keys off; **write the L5 terminal-signal semantics into L5/spawn-template.md + config.md.**

3. **The watchdog is doubly broken.** (a) **Already falsified by the repo's OWN research:** `watchdog-design-01.md` + ARCHITECTURE-FINDINGS (F-011/F-012/F-014) — the recovered phase-2-runs work — concluded timeout-kill is wrong and designed an **evidence-LEASE** replacement (healthy / stale_suspect / recovery_in_progress / failed_confirmed; lease-epoch + owner-token vs split-brain; "move to failed only on explicit evidence, not elapsed time"). Canonical docs contain **zero** lease/stale_suspect text → the build would implement the model the research already disproved. (b) **Unsanctioned kill authority:** decision #4's watchdog auto-kills a LIVE agent, but agent-lifecycle.md:27 / ARCHITECTURE.md:325 state categorically "only your parent can collapse your session," with infra-kills limited to the post-collapse reaper and L1's confirmation-gated force-reap. A blind cron watchdog reaping a live session is none of those.
   → **Resolved:** promote the evidence-lease model into canon (new WATCHDOG.md / agent-lifecycle section) as the v1 mechanism, replacing decision #4 + #6's 4-state ledger. AND resolve authority: either watchdog **detect-and-notify-parent only**, or add an explicit third sanctioned infra-kill class ("watchdog-reap of a non-signing leaf") scoped to ephemeral L5/L5+ with stated preconditions.

4. **Inferred idle-detection contradicts the corpus's "deterministic, not self-reported" promise — and the documented detector is unobservable in v1.** Decision #5 *infers* idle from noisy effect signals and self-flags "fragile"; NOTES.md:856/867/885 + OBSERVABILITY.md:22-29 assert the OPPOSITE as a settled v1 property ("deterministic, infrastructure-tracked... observable without self-reporting" via a "blocking command wrapper"). NOTES.md:869 keys Active-vs-Waiting on "LLM API calls in progress" — exactly the OTEL signal decision #5 **defers** → the documented detector can't run in v1. No `W` window anywhere; a long quiet model turn (the normal mode of an ephemeral single-turn L5) has no detector.
   → **Resolved:** demote to "deterministic for spawn/collapse/comms transitions, **inferred-with-bounded-confidence** for working/idle"; rewrite NOTES "three states" + OBSERVABILITY:22-29 onto the actual v1 floor (window_activity, node mtimes, pane_pid CPU, transcript-JSONL growth), API-call/OTEL marked deferred, drop the "blocking command wrapper" claim; add an explicit "suspected-idle while nominally Active" detector + a measured `W`.

5. **Bus transport never specified; wake mechanism undefined; ARCHITECTURE.md:156 contradicts "own bus."** The bus is described only by semantics (real-time, pub/sub, best-effort, pointer-not-payload), never by transport (file+tail? socket? Redis? inotify?) — load-bearing because "best-effort" means opposite failure modes for a socket vs a file-watch. ARCHITECTURE.md:156 names the transport "(Life-OS bus / live-session messaging)" — directly contradicting decision #10. Wake primitive unspecified: nothing says `send-keys` into a TUI prompt reliably starts a turn (trailing Enter? modal showing? half-typed buffer?). NOTES.md:870-883 "Parked = self-waking via a blocking command" has no analog in a send-keys-driven TUI.
   → **Resolved:** name the transport + delivery/liveness semantics (e.g. append-only inbox file per node; bus appends a one-line nudge + wakes via send-keys+Enter; parked coordinator re-reads inbox on the new turn); reconcile :156 with decision #10; pick one wake machine and specify the wake contract (verify a new turn started via JSONL growth, re-prod if not; defer at non-prompt modals); retire/re-spec "Parked/self-waking."

6. **The escalation-answer round-trip back DOWN to a blocked/collapsed child is unspecified.** Every escalation section is exhaustive on the UPWARD path and on what the parent DOES, but no mechanism delivers the parent's decision back down. Hard case: if the child collapsed/was reaped during the wait, the answer must reach a session that no longer exists (presumably `--resume` + delta brief — never connected to the escalation path).
   → **Resolved:** a written answer-return contract — (a) still-alive: parent writes decision into the child's escalation.md/decision.md + send-keys "decision posted, re-read, resume"; (b) collapsed: `claude --resume <uuid>` (or cold-respawn at address) with the decision as a named delta-brief payload class; binding-ledger uuid binds the answer to the right node.

7. **Human sign-off, "L1 alerts the user," and a human control surface all have CONTENT but no TRANSPORT.** (1) Human sign-off is a HARD build-gate (PLAN-ALIGNMENT-GATE Check 8) but nothing says how the human receives the package or returns a verdict; the dwell/expansion health monitor presupposes an interactive surface never defined to exist. (2) "L1 alerts the user" (~15 uses, exact text in runtime-and-model-map.md:105) names no channel — a parked L1 TUI the user isn't attached to reaches no one, and the best-effort bus "notices later" IS the failure for a human-not-watching. (3) Only control primitive is L1's force-kill — no human pause, no human kill-a-subtree (raw `tmux kill` would desync the harness-stamped ledgers), no slot to inject an escalation answer.
   → **Resolved:** name three channels — (a) sign-off transport (package file + verdict surface that flips the deliverable-ledger gate state + emits dwell/expand events); (b) an out-of-band attention channel for L1→user distinct from the bus (push: ntfy/OS-notify/Hue/email for blocking; pull-inbox for routine); (c) a human control surface (pause-subtree flag the spawner/watchdog respect; human-kill routed through the SAME harness path that stamps ledgers, never raw tmux; an answer-escalation slot).

8. **Concurrency cap is advisory-to-judgment with no admission control; "no token limits" premise is stale.** ARCHITECTURE.md:360-362 "caps" parallel instances but every mechanism is a status-board read consumed by L1/L2 judgment; the deterministic spawn script — the one chokepoint that could enforce a ceiling — is never said to check the cap. Nothing structurally blocks the (cap+1)th spawn. And the "compute is prepaid, no hard concurrency limits" premise is contradicted by the locked model: L1-L4 are ALL Opus on one Max account sharing 5h/weekly windows + per-account throughput. **Zero** 429/rate-limit/backoff/backpressure handling anywhere. Cross-runtime cost (Codex billed separately) unmodeled; single cap pools both runtimes.
   → **Resolved:** a hard **max-in-flight admission gate in the spawn script** (atomically claim a slot before launch; over-ceiling spawns queue or bounce-to-parent, never best-effort-launch); per-runtime sub-ceilings; replace "no token limits" with a usage model (defined behavior on 429/overload/usage-exhausted: backoff/queue, never silent-fail or mis-classify as a semantic agent failure; backpressure near the usage window); decide whether the cap is denominated in sessions or shared-account usage headroom.

## IMPORTANT (needed soon; not day-one-blocking)

1. **Whole-project teardown undefined** — completed coordinators sit warm forever holding tmux windows + stale ledger rows; on final acceptance L1 should trigger ordered bottom-up shutdown, pin the tree into the 2w window, close windows, move rows to a terminal "project-complete" state recovery ignores.
2. **L5+ reviewer's own lifecycle/liveness unspecified** — it can hang and nothing catches it; an L5 collapsed-pending-accept is stuck forever if L5+ dies. Resolve: L4 owns L5+ under the same watchdog, verdict file = its sign-off, respawn fresh L5+ against the frozen rubric on death (review is stateless vs spec).
3. **Subtree-quiescence roll-up over nested grandchildren undefined + races** — agent-lifecycle.md:131 "monitors only DIRECT children" contradicts decision #5's whole-subtree quiescence; a grandchild can spawn between roll-up read and kill. Resolve: a live-descendant count / max-descendant-activity in the binding row (++ at spawn, -- at collapse) + kill requires roll-up cold AND no spawn-event in last W.
4. **Race between a downward nudge and the recipient's own state transition** — send-keys has no ack; a nudge can land in a pane being concurrently collapsed (parent-kill + watchdog-kill = two killers), or on a FRESH instance that took the seat (addresses survive respawn, F35). Resolve: quiesce-then-reap interlock (one writer to the kill decision) + fresh instances ignore/re-derive nudges they can't tie to current state.
5. **Pull-visibility has no concrete read-path; direct-children vs full-subtree unreconciled** — does the parent glob every status.md under its prefix, or read a maintained roll-up (= a second source of truth)? Reconcile scope + name the read op (deterministic glob, newest-mtime first, no agent maintaining an aggregate).
6. **Context-window exhaustion of long-lived L2/L3/L4 has no recycle policy** — only L1 is covered; auto-compact silently summarizing a coordinator's transcript is an unmeasured drift surface (drift = the #1 measured thing). Resolve: proactive-recycle (handoff + cold-respawn fresh from living docs at a context-pressure threshold) + log compaction as a drift surface. (Distinct from the resume-vs-fresh knob: this is a *still-live* coordinator.)
7. **Resume-vs-fresh necro knob collides with the gate's clean-context firewall** — PLAN-ALIGNMENT-GATE.md:50 makes clean-context a HARD correctness property; full-transcript resume is exactly what's forbidden across the planning→execution seam + at reconstruction seats. Resolve: resume permitted ONLY for same-phase same-node continuity; forbidden across the gate / the seam / blind-reconstruction seats (always fresh-from-frozen-artifact).
8. **GUI/dashboard reads a superseded single status-board model** — built against Active/Parked/Waiting/Blocked, predating the three-ledger axes + tmux liveness + OTEL cost (`tmux` 0×, no cost view despite cost being a main reason to watch a 64-agent fleet); still hard-codes the dissolved review department (GUI-DESIGN.md:61, ROADMAP.md:11); "open the terminal session you want" fails for collapsed ephemeral L5s. Resolve: update read-model to 3 ledgers + tmux + OTEL cost + a live fleet board + per-level review seat + live-panes-vs-necro interaction model.
9. **Codex-TUI pane warmth (the load-bearing floor signal for the only ephemeral level) is assumed, not verified; H40 has no first spike** — V1 Codex liveness rests on an unchecked pane-warmth property of a different runtime, and L5/Codex is the most exposed to a long-quiet-turn false-idle. H40 (suppress CC's coding-assistant base prompt) remains intention-only despite the outbound-payload oracle (OBSERVABILITY.md:104-130) being exactly the instrument to prove it. Resolve: (a) first-run Codex pane-warmth measurement (else promote the Codex session-log probe to a v1 requirement); (b) an H40 spike — boot patched vs unpatched, diff captured outbound payloads, assert coding-assistant framing gone + injected identity survives to the API boundary.
10. **Prod-mid-tool-call + spinner-warm-but-wedged residual have no detection/kill path** — a send-keys prod landing mid-tool-call can corrupt the input line; a tool wedged on network/stdin is process-alive + pane-warm + CPU≈0 + JSONL-flat = reads "still working" forever. Resolve: gate send-keys on prompt-string match (only prod at the input prompt) + a wedge detector (pane warm BUT JSONL+mtime+CPU all flat for a longer W2 → kill + ledger "wedged" + escalate).
11. **Necro replay COST of growing transcripts is unbounded/unmodeled** — `--resume` re-ingests the whole transcript every wake; event-woken coordinators + the audit layer's live re-runs replay + re-bill the same big transcripts against the very budget the cap protects. Resolve: cost-keyed resume-vs-fresh (token threshold above which wake = fresh-respawn) + count replay tokens against budget.
12. **Per-interactive-session resource envelope (RAM/FD/tmux-window) unquantified** — the 20-64 cap was a coordination-quality guess made before the tmux model was locked; dozens of persistent TUIs holding RAM while idle + per-pane polling likely hits real ceilings the docs never name. Resolve: measure idle-coordinator RAM, active-worker RAM, FD count, max stable window count; derive the cap from headroom. Fold into the same first-run test as false-idle rate.

## TOP RISKS (the few most likely to sink the build)
1. The always-on harness daemon — the unacknowledged single point of failure (undefined owner, no reboot recovery, no genesis). Pin this first or nothing can be trusted to come back after a crash.
2. Liveness is built on a foundation the substrate can't deliver: docs promise "deterministic," the model concedes "inferred + fragile," and the repo's own research already falsified the encoded watchdog. Highest contradiction density; most likely to produce silent false-idle kills or undetected wedges.
3. The interactive-tmux-TUI runtime appears 0× in the docs — transport, wake, kill-authority, resource limits, and the GUI read-model all rest on a superseded abstraction.
4. Every human-machine boundary is CONTENT without a CHANNEL — sign-off, alerts, and control surface. No-silent-degradation is a promise the wiring can't currently keep.
5. Scale treated as a coordination problem, not a resource problem — advisory cap, stale "no token limits" premise, zero 429/backoff handling. A wide build is the exact load that triggers the throttling there's no defined behavior for.
